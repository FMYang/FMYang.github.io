这篇文章就记录下解决的过程。

先来看下是界面长什么样子：

如图所示，一个类似微信朋友圈的界面。

因为项目是用swift写的，所以图片加载用的是Kingfisher这个库。

定位问题

这个界面的布局很简单，一个tableview，9中不同类型的cell。啪啪啪，很快我就完成了原型设计，加载了数据。但是滑动起来发现很卡，快速滑动的时候更明显。使用FPS工具检测下，发现大概每秒只有30-40帧，快速滑动的时候内存飙到了7、8百兆，最后闪退。

检查下cell有没有复用，发现没有什么问题。开始以为是图片框架问题，换SDWebImage(4.4.2)看了下，一样的内存问题。

用Instruments检测了下，定位到问题在SDWebImage解码这行。正常大小的图片解码肯定是没有内存问题，于是打开沙盒看了下图片大小，发现大多都是宽高为1300 x 2000、2000 x 2000像素这样的大图，这里的大图是指图片宽高的像素比较大，实际图片的大小也不大，100多Kb的样子。

这里要说下图片是怎么展示的，JPEG、PNG这样的图片是压缩格式，要渲染到屏幕上，最终都要解压到位图上。位图的计算是根据图片的宽高所占的像素点来计算的，计算公式为：

1个像素占多少位（bit），与屏幕分辩率和有关，这里假设一个像素占8位，一个100KB，宽高为2000 x 2000的图片，解码之后，位图的大小为2000 x 2000 x 8bit，大约30MB.

一张图就这么大，图片一多当然内存吃紧了。

SDWebImage解决方案

SDWebImage对图片解码是根据原图的宽高来解码的，要解决内存问题首先想到是对原图进行缩放，按缩放之后的宽高解码图片。

思路有了，开始行动吧。打开SDWebImageImageIOCoder.m文件，定位到- (nullable UIImage *)sd_decompressedImageWithImage:(nullable UIImage *)image这个方法。

在解码的方法里，我们将位图大小的计算改为了100 x 100，重新编译运行下APP，发现没有还是有内存问题。

Swift命名空间

发表于 2018-02-19 | 更新于 2019-05-12 | 分类于 iOS | 评论数：

之前在使用RxSwfit的时候对.rx这个东西的一直有些疑问，不清楚实现原理，趁着现在有些空余时间，打算深入研究下，于是有了这篇文章。

通过学习，了解到rx其实就是一个命名空间的实现，关于Swift命名空间的说明，参考喵神的这篇文章。

关于命名冲突

Swift的命名空间是以模块来划分的，一个模块表示一个命名空间，我们进行APP开发是，默认添加到主target的内容是同处于同一个命名空间的。如果用Cocoapod的方式导入的第三方库，是以一个单独的target存在，不会存在命名冲突。如果是以源码的方式导入工程中，很有可能发生命名冲突。所以，为了安全起见，第三方库都会使用命名空间这种方式来防止冲突。在Objective-C上没有命名空间，一般是使用方法名前面加前缀的方式避免冲突。

自定义Tabbar

发表于 2017-07-12 | 更新于 2019-05-12 | 分类于 iOS | 评论数：

工作中经常碰到要自定义Tabbar样式，今天就来探讨下自定义Tabbar样式的二种方案

demo下载

二种方案的优缺点

方案一：
隐藏系统的Tabbar，使用自定义的View代替Tabbar
优点：使用自定义的View，tabbar的风格完全自由设计，全部为图片，图片加文字，随意组合都非常方便
缺点：tabbar隐藏和显示的动画效果很难做到与原生媲美，特别是右滑返回手势的时候，tabbar的隐藏和显示动画很难控制

方案二：
使用KVC用自定义的Tabbar替换掉系统的Tabbar
优点：tabbar隐藏和显示的动画与系统的一致
缺点：由于使用KVC的方式，万一系统的tabbar属性改名或者改为禁止使用KVC替换，该方案就会失效

CYLTabBarController的实现思路就是基于第二种方案的
