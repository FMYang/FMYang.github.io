Veröffentlicht am 2017-06-05 | Bearbeitet am 2017-07-13 | in 技术

命名

一、变量命名

  • Objective-C中变量，包括属性、实例变量和静态变量，命名过程中三者都遵循”驼峰原则”的命名规范，并且命名的时候避免出现数字编号或者无意义的命名。

二、宏定义

  • 宏定义在很多方面都会使用，例如定义高度、工具类，还有诸如文件路径。宏定义的命名，不需要传参数是固定值的，使用小写字母”k”前缀。

  • 定义尺寸类的宏，首字母使用小写字母“k”开头，后续多单词的情况下，遵循“驼峰原则”，不使用下划线。

  • 定义沙盒目录文件的宏,使用小写字母k开头，后续的单词表明宏定义的具体意义。
  • 工具类的宏

五、关于空行问题

  • .h中的空行

    • 文件说明与头文件包含(#import)之间空1行。
    • 头文件包含(#import)之间，如果需要分类区别，各类别之间空1行。
    • 头文件包含(#import)与@class之间空1行。
    • @interface与@class之间空1行。
    • 头文件{}里面，开始声明对象成员，如果需要分类区别，各类别之间空1行，不要每个成员都空行。
    • 头文件{}外，空1行书写属性，如果需要分类区别，各类别之间空1行，不要每个属性都空行。
    • 属性下面空1行开始写方法，如果需要分类区别，各类别之间空1行。
    • 方法完成后，空1行@end。
    • 如果需要声明protocol，前后空2行。
    • 通常protocol声明在@interface之前。

  • .m中的空行

    • 文件说明与头文件包含(#import)之间空1行.
    • 头文件包含(#import)之间，如果需要分类区别，各类别之间空1行。
    • 头文件包含(#import)与宏定义之间空1行。
    • @implementation和@synthesize之间空1行,如果需要分类区别，各类别之间空1行。
    • @synthesize与方法之间空1行。
    • 方法与方法之间空1行。

  • 方法里面的空行

    • 变量声明后需要空1行，如果需要分类区别，各类别之间空1行。
    • 条件、循环，选择语句，整个语句结束，需要空1行。
    • 各功能块之间空1行。
    • 最后一个括弧之前不空行。
    • 注释与代码之间不空行。 #pragma mark - 与方法之间不空行。
    • 方法根据业务的不同，使用“#pragma mark – 注释”来隔开，提高功能业务的清晰性。
    • 多个协议写的时候的空格问题，不需要换行，等系统自己换行即可。例如:

Objective-C语言所具有的规则

一、初始化

  • 在初始化方法中，不要将变量初始化为“0”或“nil”，那是多余的，内存中所有的新创建的对象(isa除外)都是0,所以不需要重复初始化为 “0”或“nil”；

二、判断两个对象是否相等

1.按照二元运算符==比较出来的结果未必是我们想要的， 因为该运算符比较的是两个指针本身, 而不是其所指向的对象；
2.应该使用NSObject协议中声明的isEqual:方法来判断两个对象是否相等;
3.字符串使用isEqualToString:方法,比isEqual:方法快， 因为后者需要需要执行额外的操作（判断受测对象的类型）。
4.数组比较使用isEqualToArray:方法；
5.字典比较使用:isEqualToDictionary:方法。

三、BOOL陷阱

1.将int值转换为BOOL时应特别小心，避免直接和YES比较 。
2.Objective-C中，BOOL被定义为unsigned char，这意味着除了YES(1) 和 NO(0)外它还可以是其他值。禁止将int直接转换为BOOL。 
3.常见的错误包括:将数组的大小、指针值或位运算符的结果转换为 BOOL,因为该BOOL值的结果取决于整型值的最后一位
4.将整型值转换为BOOL的方法:使用三元运算符返回YES / NO。 
5.BOOL、_Bool和bool之间的转换是安全的,但是BOOL和 Boolean间的转换不是安全的,所以将Boolean看成整型值。

代码注释

一、哪里需要注释：

  • 在某个变量、属性、或者方法不能够直接从名字得知其用途时。
  • 区分代码区时（在常量文件和控制器类的.h文件和.m文件里）。
  • 在不相关的业务处理交叉点。
  • 文件的基本信息
  • 有疑惑的地方（FIXME）
  • 未完成或者待优化的地方。（#waring）
  • 发布上线要修改的地方。(//TODO:)
  • 声明枚举时应该对每个值说明。
  • 修改别人代码的时候。
  • 编写API时，一般需要对所有的接口带上注释

二、注释需要注意：

  • 不要为了注释而注释，请只在关键点注释
  • 划分代码块最好的方式不一定是注释，有时候使用空行也可以达到很好的效果。

三、注释方式：

  • 单行注释
    // 简要说明（在Xcode中对属性的注释请使用这个）

  • 多行注释

    / … /

  • 行尾注释

    @property (nonatomic, strong) NSString *userName; // 用户名

  • 标识注释

四、在oc文件中的注释

  • 类注释

    • 类声明的类名上面加上一段对该类的说明。

  • 方法注释

    • 如果是API中的方法，那么只要是暴露到头文件的方法都必须加上注释。
    • 如果是类文件的方法，则可以对关键的方法给出合理的注释。
    • 方法的注释请使用多行文本注释方式。

  • 属性注释

    • 属性的注释方式，一般有两种：尾行双斜杠注释和单行双斜杠注释的方式。
    • 尾行注释一般用于.m文件的私有属性的注释，单行注释一般用于.h文件的公有属性的注释。

  • 逻辑注释

    • 在程序较复杂处理的地方，应该对关键的步骤加上注释，一般使用单行注释。
    • 如果可以用换行解决的就用换行代替就可以了。

  • 常量定义和宏定义注释

    • 使用单行双斜杠和尾行注释皆可，看情况

  • 枚举注释

    • 对枚举声明的枚举加上一个单行注释。
    • 对枚举的各个枚举值都加上尾行注释。

一、单例的使用

  • 单例模式是一种常用的软件设计模式。在它的核心结构中只包含一个被称为单例类的特殊类。通过单例模式可以保证系统中一个类只有一个实例而且该实例易于外界访问。

  • 在IOS中使用单例模式的情况

    如果创建一个对象会耗费很多系统资源，那么此时采用单例模式，需要一个实例，会节省alloc的时间。

二、init初始化方法的使用

  • init继承于NSObject这个根类，所有的子类可以不用重写这个实例方法函数，当然也可以在自己的类里重写init实例方法,把这个方法放在- (void)delloc方法之后。
  • 生命周期业务（life cycle）
    • (void)delloc;
    • 懒加载部分
    • (void)init;
    • (void)loadview;
    • (void)viewDidLoad;
    • (void)viewWillAppear;
    • (void)viewDidAppear;
    • delegate和业务方法
    • (void)didReceiveMemoryWarning;
init方法里不要出现创建view的代码。良好的设计，在init里应该只有相关数据的初始化，而且这些数据都是比较关键的数据。init里不要调self.view，否则会导致viewcontroller创建view。（因为view是懒加载的）。

loadView方法中只初始化view，一般用于创建比较关键的view如tableViewController的tabView，    UINavigationController的navgationBar，不可调用view的getter（在掉super loadView前），最好也不要初始化一些非关键的view。

viewDidLoad方法这时候view已经有了，最适合创建一些附加的view和控件了。

viewWillAppear方法一般在view被添加到superview之前，切换动画之前调用。在这里可以进行一些显示前的处理。比如键盘弹出，一些特殊的过程动画（比如状态条和navigationbar颜色）。

viewDidAppear方法一般用于显示后，在切换动画后，如果有需要的操作，可以在这里加入相关代码。

didReceiveMemoryWarning方法是在内存警告时，会调用这个方法，所以一般设置的delegate,在这个方法里面将它设置为nil;

三、iOS的沙盒访问

  • Documents:应用中用户数据可以放在这里，iTunes备份和恢复的时候会包括此目录。
//获取目录的方法 
NSArray *paths = NSSearchPathForDirectoriesInDomains(NSDocumentDirectory, NSUserDomainMask, YES); 
NSString *docDir = [[paths objectAtIndex:0] stringByAppendingString:@"/Documents"];
  • tmp:存放临时文件，iTunes不会备份和恢复此目录，此目录下文件可能会在应用退出后删除。
NSArray *paths = NSSearchPathForDirectoriesInDomains(NSDocumentDirectory, NSUserDomainMask, YES); 
NSString *docDir = [[paths objectAtIndex:0] stringByAppendingString:@"/tmp"];
  • Library/Caches:存放缓存文件，iTunes不会备份此目录，此目录下文件不会在应用退出删除。 
NSArray *paths = NSSearchPathForDirectoriesInDomains(NSDocumentDirectory, NSUserDomainMask, YES); 
NSString *docDir = [[paths objectAtIndex:0] stringByAppendingString:@"/Library"];
